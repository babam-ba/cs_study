# 라이브러리, 프레임워크
> 라이브러리와 프레임워크에 대해서 설명하세요.⭐️⭐️

## 공통점
1. 재사용 가능한 코드의 모음
2. 프로그래밍을 쉽게 할 수 있도록 도와주는 역할

## 차이점
### 프레임워크
일정한 형태와 필요한 기능을 갖추고 있는 뼈대를 의미하며 정해진 매뉴얼, 룰을 제공한다. 개발 시 필수적인 코드, 알고리즘 등 기능을 제공하는데 이 룰을 지켜야 기능을 사용할 수 있다.

ex) Vue, Angular, Next.js 등

### 라이브러리
특정한 기능을 구현하기 위해 미리 만들어진 코드들의 집합이다. 필요할 때만 자유롭게 사용할 수 있는 도구이다.

라이브러리를 사용하더라도 프레임워크가 정한 방식에 따라야 하지만 어떻게 사용할 지는 개발자가 정할 수 있다.

Next.js와 React의 예를 들자면 React는 UI를 만들기 위한 라이브러리이다. Next.js는 React를 사용하여 UI를 만들고 Image 컴포넌트를 사용하여 이미지를 최적화하며 pages라는 파일 시스템을 제공하여 라우팅을 쉽게 구현할 수 있도록 도와준다. 또한, SSR을 지원하는 등 이미 주어진 재료들이 있고 정해진 룰을 지키면 이 기능을 사용할 수 있는 것이다.

ex) React, Redux, jQuery 등

### 제어 흐름의 권한
라이브러리를 사용할 때 개발자는 애플리케이션 코드의 흐름을 **직접 제어**해야 한다. 개발 시 능동적으로 원하는 코드를 라이브러리로부터 불러와 사용하는 것이다.

반면 프레임워크는 애플리케이션 코드가 프레임워크(뼈대)에 의해 제어된다. 프레임워크가 정해놓은 규칙(틀)에서 **수동적으로 동작**하기 때문에 제어의 흐름은 프레임워크가 가지고 있고 개발자는 정해진 뼈대 안에서 필요한 코드를 작성한다.

### IoC(Inversion of Control, 제어의 역전)
어떠한 일을 하도록 만들어진 프레임워크에 제어 권한을 넘김으로써 client 코드가 신경써야 할 것을 줄이는 전략.

오브젝트 생성, 관계 설정, 사용, 제거 등 오브젝트 전반에 걸친 모든 제어권을 애플리케이션이 갖는 것이 아니라 프레임워크의 컨테이너에게 넘기는 개념이다.

#### 장점
- 개발자는 특정 기능에 덜 신경쓰게 되어 다른 부분에 더 집중할 수 있다.
- low coupling을 이용해 객체간 의존 관계를 쉽게 변경할 수 있다.(구현, 수행을 분리하기 때문)
  -> 궁극적으로 코드의 재사용성, 유지보수성 상승

#### 단점
- 역제어 구조로 되어 있기 때문에 코드를 이해하기 어려울 수 있다.

### DI(Dependency Injection, 의존성 주입)
IoC의 구현 방법 중 하나로, IoC 원칙을 실현하기 위한 여러 디자인 패턴 중 하나이다.
객체 간 결합을 느슨히 만들어 유연하고 확장성 뛰어난 코드를 작성하기 위한 패턴

종합적으로 봤을 때, 라이브러리의 경우 애플리케이션의 흐름을 사용자가 직접 제어해야 하지만 프레임워크의 경우 코드를 연결할 수 있는 위치를 제공하고 필요에 따라 사용자가 연결한 코드를 호출하는 제어 흐름 권한을 가지고 있다.
